# 研究范围：跨服务端—客户端的 LLM 流式工具调用兼容性

版本：0.1  
冻结日期：2026-07-18

## 问题陈述

大量 LLM 推理服务宣称兼容 OpenAI API，但“请求能够成功返回”并不意味着它可以被真实客户端无缝替换。工具调用以 SSE 增量传输时，调用 ID、工具名、参数 JSON 和结束状态会分布在多个事件中。即使每个事件都满足 JSON Schema，不同的分片边界、可选字段、事件顺序及并行调用交错方式，仍可能触发客户端的隐含假设，造成参数丢失、调用合并、解析异常或无限等待。

本研究将兼容性定义为一个关系：给定同一项工具调用语义，服务端产生的合法流被客户端消费后，应恢复出与 ground truth 等价的规范化工具调用，而不是仅通过单个响应对象的 Schema 校验。

## 研究问题

- **RQ1：** 主流 OpenAI-compatible 推理服务与客户端 SDK 的流式工具调用兼容性缺陷有多普遍，主要表现为何？
- **RQ2：** 语义保持的流式轨迹变换能否发现传统示例测试和 Schema 合规测试遗漏的缺陷？
- **RQ3：** 使用客户端最终恢复结果的语义预言，能否准确区分无害格式差异与会破坏应用的行为差异？
- **RQ4：** 自动缩减能否把失败轨迹转化为稳定、可由上游维护者确认的最小复现案例？

## 明确纳入与排除

| 纳入第一阶段 | 排除第一阶段 |
| --- | --- |
| `/v1/chat/completions` | `/v1/responses` 及其他端点 |
| `stream=true` | 非流式接口的一般一致性 |
| 单个和并行 `tool_calls` | 文本回答质量与模型能力排名 |
| SSE 分片、顺序、结束和中断 | 吞吐、延迟、显存等性能指标 |
| 客户端解析后的工具语义 | 多模态、音频、鉴权与计费 |
| 服务端和客户端的版本差异 | 提示词鲁棒性与 Agent 规划能力 |

## 方法边界

系统首先从人工构造或真实服务记录得到规范轨迹，并保存规范化 ground truth：`(index, call_id, name, arguments_json, finish_reason)`。生成器对轨迹实施语义保持变换，包括参数字符串拆分/合并、空增量插入、可选空值变化以及并行工具调用的合法交错。轨迹由可控 Mock Server 回放给真实客户端 SDK；各适配器将客户端结果还原到统一表示，并与 ground truth 比较。

失败包括：异常退出、超时、工具数量变化、调用身份混淆、工具名改变、参数 JSON 无法解析或语义改变、结束状态误判。测试不比较模型自然语言输出，也不把“多数实现一致”直接当作正确答案。

第二阶段才从 llama.cpp、Ollama、vLLM 和 SGLang 记录真实流，并形成“服务端版本 × 客户端版本 × 轨迹类别”的实验矩阵。研究贡献必须包含方法效果对比、缺陷去重和维护者确认，不能只报告兼容率。
